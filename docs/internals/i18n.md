# i18n

Supported locales: `ja` (default), `en`. Backend uses Laravel's `__()` / `@lang()`;
the Inertia/React frontend uses [`laravel-react-i18n`](https://github.com/xiCO2k/laravel-react-i18n).
React reads `lang/*.json` via Vite's `import.meta.glob`; the package's Vite plugin
compiles `lang/{locale}/*.php` namespaces into a transient `lang/php_{locale}.json`
(gitignored) that the same glob picks up, so PHP dotted keys (`block.confirm_block`)
resolve on **both** sides.

## Translation key convention

Two key styles coexist:

- **Structured keys** for app UI strings вҖ” a per-feature PHP namespace,
  `<feature>.<item>` in snake_case (`block.confirm_block`, `block.member_id`).
  This is the convention for new app strings, and existing flat strings migrate
  to it feature by feature.
- **English source text** for cross-cutting / generic strings still in the flat
  dictionary (`__('Cancel')`). The framework renders the English key verbatim
  when no entry exists, which the omission policy below exploits.

The flat English-source dictionary shrinks as features move to structured keys;
the end state is `lang/{ja,en}.json` holding publisher keys only.

## File layout

App UI strings live in **per-feature PHP namespace files** `lang/{ja,en}/<feature>.php`
(e.g. `block.php` вҶ’ `block.confirm_block`). Name the namespace by feature/domain,
not by UI layout; keep keys shallow (`<feature>.<item>`, occasionally
`<feature>.<group>.<item>`); the leaf is snake_case and names the string's role,
not its English text. One key per UI instance вҖ” do not reuse a key across
contexts where the translation could diverge.

Cross-cutting / generic strings (`Cancel`, `Save`) that translate identically
everywhere may stay in the flat `lang/ja.json` + `lang/en.json` (English source
text as key) until a `common` namespace is warranted. All dictionaries are
eager-glob-merged into one bundle at build time, so splitting is for ownership
and clarity, not lazy loading.

Framework and reference data also use PHP namespace files: `validation`, `auth`,
`passwords`, `pagination`, `http-statuses`, `actions` (publisher-managed),
`terms`, `regions` (app-authored) вҖ” see [Ownership](#ownership-publisher-managed-vs-app-authored).

`lang/{locale}/` holds flat `*.php` group files only вҖ” **no subdirectories**.
Express hierarchy with nested PHP arrays. The laravel-react-i18n Vite parser
recurses subdirs into `dir.file.key` keys, but Laravel's backend loader reads
only top-level `{group}.php`, so a subdirectory would resolve on the frontend
and miss on the backend (`i18n:check` fails on any subdirectory).

## Ownership: publisher-managed vs app-authored

`lang/` is committed to this repo, but two parties write into it:

- **publisher-managed**: the laravel-lang entries inside `lang/{ja,en}.json`
  (English-source keys it ships) **plus** the framework PHP namespaces `validation`,
  `auth`, `passwords`, `pagination`, `http-statuses`, `actions`. `lang:add` /
  `lang:update` write these; do **not** hand-edit them вҖ” the next `lang:update` merges
  publisher values back over the edit. A `package::group.key` PHP namespace overrides
  under `lang/vendor/{package}/{locale}/...`; the JSON source keys and core groups
  published into the root have no `lang/vendor` override, so a durable override needs a
  dedicated override path, not an in-place edit.
- **app-authored**: OpenPNE's own PHP namespaces and, today, the app's own keys in
  `lang/{ja,en}.json`. Split in two by coverage rule:
  - **app-ui** (`terms`, `block`, and per-feature groups as strings migrate):
    structured keys with no "key === English text" fallback, so every key needs a
    real value in **both** ja and en (`i18n:check` enforces it).
  - **app-reference** (`regions`): master data that tolerates source fallback / partial
    coverage (`lang/en/regions.php` is empty; `RegionListService` falls back to the
    English source name). Exempt from the en+ja requirement; still boundary-checked.

`i18n:check` keys these zones off **group classification** (`PUBLISHER_GROUPS` /
`APP_UI_GROUPS` / `APP_REFERENCE_GROUPS` in `CheckTranslationsCommand`), not a prefix.
An unclassified PHP group fails the gate вҖ” so a group newly published by `lang:update`,
or a misplaced app file, gets a deliberate home rather than silently joining the catalog.

`lang/{ja,en}.json` therefore mixes app keys with publisher keys (e.g. pagination
`to`/`results`/`Showing`/`of`). Several publisher keys are rendered only by
framework/vendor code (the pagination Blade, validation, http-statuses), so they do
**not** appear in the app-code scan.

> **`i18n:check --unused` is informational, NOT a deletion list.** It reports JSON
> keys the app-code scan does not reference, but that set includes publisher keys the
> framework renders at runtime. Removing them breaks framework output вҖ” this is how
> `to`/`results` were once deleted, turning pagination into "X to Y of Z results".
> Restore via `lang:update` (or by hand for one or two keys); never treat `--unused`
> as a delete list.

## Key ordering

`lang/ja.json` / `lang/en.json` keys are kept in a canonical order вҖ” ASCII
case-insensitive with an uppercase-first byte tiebreak, lexicographic (not
numeric-aware, so `Page 10` precedes `Page 2`). Every key has one deterministic
insertion point, which removes merge churn and the "where does this go?" guesswork.

- `php artisan i18n:check --sort` rewrites both files in order (`--sort=lang/ja.json`
  scopes to one file). The default `i18n:check` run fails when a file is out of order.
- The pre-commit hook auto-sorts staged `lang/{ja,en}.json` and re-stages them. A
  partially staged dictionary is refused (sorting would also stage the unstaged
  hunk) вҖ” run the printed `--sort=<file> && git add <file>` instead.
- `i18n:check` prints a non-failing advisory of case-fold key collisions (e.g.
  `Show all` vs `Show All`) so inconsistent translations surface. Intentional
  collisions with distinct meaning per case are recorded in
  `lang/.i18n-collision-allowlist.json` (matched on the exact key set) and suppressed.

## Key specificity (semantic collisions)

A key whose correct translation depends on unstated context is a latent bug:
the coverage gate cannot catch it (the key exists), so a wrong reuse вҖ” e.g.
`__('From')` rendering е·®еҮәдәә ("sender") on a date range вҖ” ships silently.

- **Make keys specific, not short.** The risk is *polysemy* (a word needing
  different translations in different contexts), not length. A short generic
  key is fine when its translation is invariant everywhere (`Name`вҶ’еҗҚеүҚ,
  `Save`вҶ’дҝқеӯҳ) вҖ” do not over-specify those, it duplicates values and
  reintroduces drift. Disambiguate only the polysemous ones: `Sender` /
  `Recipient` (not `From` / `To`), `Member count` (not `Members`), `Main area`
  (not `Contents`).
- **Prefer a structured key; disambiguate within the flat dictionary.** A
  per-feature structured key (`block.member_id`) is inherently context-scoped, so
  polysemy rarely arises. For strings still in the flat dictionary, choose a more
  specific English source key and, when the English display must differ from the
  key, add the `en.json` value in the same change.
- **True homographs** вҖ” one English word with two senses that must both render
  that word вҖ” take a parenthetical tag from the closed set `(noun)` / `(verb)`
  / `(adjective)` / `(adverb)`, e.g. `Post (noun)` / `Post (verb)`, with
  explicit `en` and `ja`. Try rephrasing to an unambiguous key first.
- **Reuse before adding.** Check for an existing key with the same meaning
  before adding a near-duplicate (`Order` was folded into `Sort Order`).
- **Keep translatable strings in the catalog.** Route every user-facing string
  through `__()` / `t()`; do not inline `['ja' => вҖҰ, 'en' => вҖҰ]` arrays in
  code. Punctuation that is part of the rendered phrase belongs inside the
  value, not concatenated around the call.
- **Do not pre-seed generic keys** without a call-site; add a key when you add
  its use, translated for that use.

`i18n:check` backs these: a non-failing **near-fold** advisory flags label keys
whose singular/plural stem matches but whose Japanese differs (e.g.
`Link`/`Links`), and `--duplicates` lists keys that share one Japanese value
(consolidation candidates). Intentional near-fold and case-fold pairs are
recorded in `lang/.i18n-collision-allowlist.json`. Marker keys must carry a real
(non-identity) `en` and `ja` value вҖ” a hard gate (see [CI gate](#ci-gate)).

## Omission policy ("key === text" pruning)

Both `__()` and `laravel-react-i18n` return the key verbatim when a key has no
entry. So entries where `key === value` are redundant.

| File | Pruning rule |
|------|--------------|
| `lang/en.json` | All `key === value` entries removed. English text is the key itself, so the fallback renders correctly. |
| `lang/ja.json` | Pre-translated keys like `"OK": "OK"` stay (valid Japanese that happens to match its key). The one exception is **pure-placeholder keys whose names are configured terms** (`%Friend%`, `%Communities%`, see [Term placeholders](#term-placeholders)) вҖ” the term layer renders those without a JSON entry, so identity rows are redundant. A typo like `%Firend%` is *not* exempt and fails the coverage check. |

`php artisan i18n:check --prune-identity` enforces both rules вҖ” pruning every
identity row from `en.json` and only pure-placeholder identity rows from
`ja.json`.

## Locale resolution

[`SetLocale`](../../app/Http/Middleware/SetLocale.php) is registered on
**both** the `web` middleware stack (`bootstrap/app.php`) and the Filament
admin panel stack (`AdminPanelProvider::panel()`) вҖ” the Filament panel does
not inherit `web`, so it needs its own entry. It runs in priority order:

1. `session('locale')` вҖ” set by `POST /locale` (`route('locale.switch')`)
2. `Accept-Language` header negotiated against `['ja', 'en']`

`Accept-Language` falls back to the first supported locale (`ja`) when there is
no preference. Member-level persistence is not wired yet; add it as step 3 here
once a member-preference store exists.

## Inertia / React wiring

- [`vite.config.ts`](../../vite.config.ts) registers the `laravel-react-i18n/vite`
  plugin, which compiles `lang/{locale}/*.php` into a transient `lang/php_{locale}.json`
  on `dev`/`build` (cleaned at build end, gitignored). The provider's glob then merges
  JSON **then** `php_*.json`, so a PHP-namespace key wins on collision (guarded by the
  CI gate below).
- [`HandleInertiaRequests::share()`](../../app/Http/Middleware/HandleInertiaRequests.php)
  exposes `locale` so the React provider can read it from the initial page props.
- [`app.tsx`](../../resources/js/app.tsx) wraps the Inertia app in
  `LaravelReactI18nProvider` with `fallbackLocale="en"`. Using `en` as the
  fallback (not the app default `ja`) ensures an en miss surfaces as the raw
  English key вҖ” otherwise the empty en bundle would silently render Japanese.
- [`useT()`](../../resources/js/lib/i18n.ts) wraps `useLaravelReactI18n().t`
  and applies the term substitution (see [Term placeholders](#term-placeholders)).
  Components should always import `useT` from `@/lib/i18n` rather than the
  package directly; reading `t` straight from the package skips the
  substitution and breaks `%Friend%`-style placeholders.
- `LaravelReactI18nProvider` reads `locale` from `initialPage.props` at app
  boot and does not subscribe to subsequent Inertia prop updates. `POST /locale`
  therefore returns `Inertia::location()` for Inertia requests, which makes
  the client do `window.location = url` and remount the provider. A future
  React-side switcher MUST trigger a hard navigation the same way (Inertia
  `router.post('/locale', ..., { preserveState: false })` is not enough on
  its own).

## CI gate

`php artisan i18n:check` scans `resources/js/`, `resources/views/`, `app/`,
`database/`, `routes/`, `config/` for `t('...')` / `__('...')` / `@lang('...')`
single-line literals and fails when any key is missing from `lang/ja.json`
(modulo `lang/.i18n-baseline.json` grandfathering вҖ” only created on first need
via `--update-baseline`). Dynamic keys (`t($var)`) are silently skipped. It also
fails when `lang/{ja,en}.json` is not in canonical key order (see [Key ordering](#key-ordering)),
or when a `(noun)`/`(verb)`/`(adjective)`/`(adverb)` marker key lacks a real
(non-identity) translation in `ja.json`/`en.json` (see [Key specificity](#key-specificity-semantic-collisions)).

It also enforces the **namespace boundary** between JSON and PHP groups (hard
fails, see [Ownership](#ownership-publisher-managed-vs-app-authored)):

- a flat JSON key may not fall under a PHP group (`terms.x`, or the bare group
  `terms`) вҖ” laravel-react-i18n merges `php_{locale}.json` last, so the PHP value
  would silently win over the JSON key;
- every PHP group must be classified (publisher / app-ui / app-reference);
- `lang/{locale}/` must contain no subdirectories;
- app-ui group keys must exist in both ja and en.

- [`tests/Feature/I18nCoverageTest.php`](../../tests/Feature/I18nCoverageTest.php)
  runs the command as part of `php artisan test` so the gate triggers on every
  test run.
- [`tools/git-hooks/pre-commit`](../../tools/git-hooks/pre-commit) blocks
  commits that touch `*.tsx|*.ts|*.php|*.blade.php` when new gaps appear.
  Install once with `composer install-hooks`.

## Term placeholders

Administrators can rebrand SNS terms (`friend`, `community`, `nickname`, ...)
without changing code. Use placeholders in user-facing strings:

| Form | Resolves to | Example |
|------|-------------|---------|
| `%name%` | Configured value, lowercase | `%friend%` вҶ’ `гғ•гғ¬гғігғү` / `friend` |
| `%Name%` | First letter capitalised in `en`; identical in `ja` | `%Friend%` вҶ’ `гғ•гғ¬гғігғү` / `Friend` |
| `%names%` | Plural form derived in `en`; identical in `ja` | `%communities%` вҶ’ `гӮігғҹгғҘгғӢгғҶгӮЈ` / `communities` |
| `%Names%` | Capitalised + plural | `%Friends%` вҶ’ `гғ•гғ¬гғігғү` / `Friends` |

Use a placeholder even if the configured value happens to match the literal
("friend" in en defaults). Writing the literal opts out of admin
customisation silently.

**Defaults** ship in `lang/{ja,en}/terms.php`. The key set defined there
is the entire surface the admin form exposes вҖ” adding a term means adding
a key in both files (the `TermDefaultsConsistencyTest` invariant enforces
matching key sets, no schema migration needed).

**Overrides** live in the `term_overrides` table. The admin page at
`/admin/term-settings` only persists rows that diverge from defaults:
blank inputs and inputs matching the default delete the row. Avoid writing
code that seeds rows matching the default вҖ” the resolver does
`array_merge(defaults, overrides)`, so redundant rows are silent dead
weight.

**Variants are server-side**. `HandleInertiaRequests::share()` expands
each term into the four `%name%/%Name%/%names%/%Names%` variants before
shipping the `terms` prop. The client is a flat dictionary lookup; do
not add case or plural logic in React.

**Backend hooks**. `__()` is routed through `TermTranslator`, which
also overrides `has()` so `Lang::has` / `trans_choice` keep their
existence semantics (a key consisting only of `%name%` tokens reports
`false` from `has()` even though `__()` renders it).

## Key invariants

1. `lang/en.json` MUST NOT contain `key === value` entries, nor MUST
   `lang/ja.json` contain pure-placeholder identity rows вҖ” always run
   `--prune-identity` after `lang:add en` / `lang:update` or other bulk imports.
2. Every code-referenced key MUST have a `ja` value, except pure-placeholder
   keys whose names are configured terms (e.g. `%Friend%`). Typos like
   `%Firend%` are flagged by the coverage check.
3. `POST /locale` MUST trigger a hard navigation for Inertia requests so the
   React provider remounts вҖ” see the Inertia / React wiring section above.
4. Term names (`friend`, `community`, ...) MUST be referenced via `%name%`
   placeholders, never as literal strings. Adding a new term name MUST add
   the key to **both** `lang/ja/terms.php` and `lang/en/terms.php`.
5. `lang/{ja,en}.json` MUST be in canonical key order вҖ” run
   `php artisan i18n:check --sort` (the pre-commit hook and CI gate enforce it).
6. A flat JSON key MUST NOT fall under a PHP namespace group, every PHP group
   MUST be classified, and `lang/{locale}/` MUST stay flat (no subdirectories) вҖ”
   see [Ownership](#ownership-publisher-managed-vs-app-authored) and the CI gate.
